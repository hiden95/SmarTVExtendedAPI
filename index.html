This document introduces Smart TV Extended APIs. These APIs are defined in "HTML5 Based Smart TV Platform"(TTAK.KO-07.0111/R1) to control Smart TV specific features like channel, program, device and so on. The specification has been standardized for common platform of Smart TV based on W3C web runtime environment to make an application running on various devices with the same user experiences. Smart TV Extended APIs consist of application interfaces, broadcast interfaces, device interface, multiscreen interfaces and DRM interfaces. These interfaces can be accessed as attributes of window object that is used globally in web environment.

Introduction

"HTML5 Based Smart TV Platform" specifies a common platform on which applications developed based on HTML5 run in various broadcasting environments such as terrestrial, cable, satellite and IPTV. An application in compliance with this standard is developed and released taking advantage of HTML5-based technologies and interfaces and will provide the same user experience on various devices. This standard proposes the generalized technology criteria for development, distribution and execution of HTML5 based smart TV platform's applications.

The followings are key features of this specification.

Application Model/Life cycle

Smart TV's application types are defined by the following criteria.

Execution Method

TypeDescription
store appThis application is launched(or ended) by User after that is downloaded and installed on the Smart TV device through App Store(or Market)
signal appThis application is launched(or ended) by Application Information Signal(AIT) of broadcasting service provider.
broadband appThis application is launched(or ended) by API call defined in Smart TV Extended APIs.

App Packaging

TypeDescription
package appAll application resources are compressed to specific packaging format. This application is delivered and installed through App Store.
non-package appThis application is accessed by application URL.

Broadcasting Relation

TypeDescription

broadcast inactivated appWhen this application run, the broadcasting resources shall be suspended. Application is running alone without broadcast service

Channel Bound

TypeDescription
channel bound appWhen TV channel is changed, this application is ended automatically.
channel unbound appWhen TV channel is changed, this application is still running in the new channel.

Application Life Cycle

The cases that any application can be started are like below.

  • User select store app that is already installed on the Smart TV device.
  • API call by the current running application. (createApplication)
  • Application singal(AUTOSTART) in broadcasting stream made by broadcasting service provider.

On the other hands, the cases that application can be ended are like below.

  • Channel change while channel-bound application is running.
  • API call by current running application. (destroyApplication)
  • Application signal(KILL) in broadcasting stream made by broadcasting service provider.
  • Application priority (Low priority application shall be ended when higher application is started.)
  • EXIT key on remote controller is pressed by User.
  • Smart TV exceptions or unexpected cases.(ex. system memory full)

Application Signaling

In this specification, application signaling definition is based on [[DVBSIGNAL]]. The definition of "application_signaling descriptor" that is used in PMT of broadcasting stream is complied with [[DVBSIGNAL]]'s 5.3.5.1.

And the followings among AIT descriptors defined in [[DVBSIGNAL]] are supported in this specification.

  • application descriptor
  • application usage descriptor
  • transport protocol descriptor
  • simple application location descriptor
  • application name descriptor

As application delivery protocol, this specification support HTTP only and it doesn't support application delivery through DSMCC.

Application Packaging

All applications distributed and installed through App Store shall be compressed and packaged by specific format. The package requirements of this specification is based on [[widgets]]. Additionally, this specification introduces some features and element like below.

This specification defines new URL scheme to use it as "id" attribute's value of widget element.

    tvapp://widget.organization_id.application_id

All applications complied with this specification shall have id as the pair of two values to identify it; organization_id and application_id. So, the package application shall assign the value to as id attribute of widget element according to URL scheme above.

And the following elements are defined newly in this specification.

W3C profile for Smart TV

This specification supports application execution based on W3C web environments. However, considering Smart TV device's features like screen resolution and input method, some W3C specifications are not suitable to TV environment, so we need to make W3C profile for TV. To do this, this specification defines W3C profile based on [[OIPFTVPROFILE]]

[[OIPFTVPROFILE]] shows a profile definition of HTML, CSS and other related web technologies aimed to TV services and devices. Its goal is to describe a common profile that can be relied on by content and service providers and implemented by manufacturers.

Additionally, this specification profiles the following definitions.

  • Virtual KeyCode Definition : It does not follow the "Virtual KeyCode Constants" of [[OIPFTVPROFILE]] 6.2
  • Geolocation API supported
  • Media APIs profile for handling broadcasting video

Receiver Minimum Requirements

To be complied with this specification, Smart TV device shall meet the following minimum requirements.

RequirementDescription
Screen Resolution960 X 540(at least), 1280 X 720(recommended)
Application TransparencyTV devices supported transparency completely. It shall make user not to know the difference between original broadcasting video and the same video behind the transparency application.
Color Format for Graphic PlaneRGBA16(at least), RGBA32(recommended)
Text Input MethodTV device shall have the method for text input like IME or virtual keyboard.(specific method is not defined)
Video scalingArbitrary video scaling shall be supported between 1/4 X 1/4 and 2 X 2. The recommended minimum size is 1/8 X 1/8.
Video aspect ratioThe original aspect ratio of decoded video shall be preserved when the video is scaled.
FontThis specification does not define the default font. The device shall support web download font.
Runtime memoryThe minimum runtime memory supported by device is 30MBytes.
Input Device The receiver shall support the remote controller by default. When the button on remote controller is pressed, the device shall occur key event using virtual key code defined in this specification.(ex.VK_RED, VK_UP, VK_ENTER, VK_9 and so on)
And the pointing device like mouse is recommended.

Receiver Profiles

This specification defines profile about itself considering the device's main services based on device capabilities, broadcasting environments, business model and so on. The profile includes implementation coverage and the specific restrictions for this specification. the profile definition is like below.

ProfileDescription
Full ProfileThe receiver of this profile supports all requirements of this specification.
Broadcast Related ProfileThe receiver of this profile just support the requirements for the application launched by broadcasting signal.
So, pakcage application is excluded in this profile.
Basic Package ProfileThe receiver of this profile just support the limited package application. The package application with broadcast activated feature is not included this profile.
And the signal application launched by broadcastor is also excluded in this profile.
Advanced Package ProfileThe receiver of this profile supports the full package application.
However, the signal application launched by broadcastor is excluded in this profile.

Handles the onChannelChanged event of type ChannelChanged, fired when Smart TV's channel is changed by API call or remote controller.
attribute EventHandler onProgramChanged
Handles the onProgramChanged event of type ProgramChanged, fired when the broadcasting program is changed.

In this specification, the channel change operation can be done by HTML5 <video> element like below

			var video = document.querySelector('video');
			var chMgr = tvExt.broadcast.channelManager;
			video.src = chMgr.getCurrentChannel().sourceURI; // current channel is binding to HTML5 video element
			
			video.src = chMgr.getNextChannel().sourceURI; // channel changed to next channel

DrmAVObject Interface

This interface is defined to handle DRM-related information of object playing DRM-protected contents. Even if the play itself uses HTML5MediaElement, the application can handle DRM-related APIs without any extension of HTML5MediaElement.

DOMString DRMSystemID(HTMLMediaElement object)
This method is called to get the ID value of DRM system that is adapt to HTML5 object indicated by parameter.
HTMLMediaElement object
the object instance that is playing DRM protected contents currently.
attribute EventHandler onDRMRightError
Handles the onDRMRightError event of type DRMRightError, fired when received error message from DRM agent during the play.
